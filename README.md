# UT Manager

> 成果物ではなく「コードを書くプロセス」を見て、エンジニアの本当の実力を見抜き、マネージャーの人員配置とプロジェクト遂行を支えるAI。使うほど各人にパーソナライズされて育つ。

**DevOps × AI Agent Hackathon 2026**（主催：ファインディ／メインスポンサー：Google Cloud／協賛：Elasticsearch）提出作品。テーマ「つくる。まわす。とどける。」

🔗 **本番デモ**：https://ut-manager-759497786842.asia-northeast1.run.app

## 課題

AIの普及で「手を動かせる人」は激増した一方、理解を伴う本物のエンジニアはむしろ見えにくくなった。GitHub等の成果物は嘘をつける（コピペも自力もコミットされれば同じ顔）。理解度は完成コードだけでは測れない。2026年1月の研究では、行動ログだけでは「人がキーボードを操作したこと」までしか分からず「中身を人が考えたか」は判定できないと証明された。

見るべきは「どう手を動かしたか＝プロセス」。プロセスは嘘をつけない。そして理解を測るには「なぜこうした?」と思考そのものを問うしかない。

## コアコンセプト

1. **プロセスを見る** — 打鍵・ペースト・手戻り・停止・実行回数・エラー履歴を記録。成果物でなく過程で判断する。
2. **事実で測る** — 自己申告を信じず、行動シグナルと「なぜこうした?」への回答の深さで実力を測る。
3. **育つ** — 各ユーザーの履歴（盛る／隠す癖、詰まりやすい領域）を蓄積し、使うほど判定と指導が個別最適化される。

## プロダクト全体像（2フェーズ）

DevOpsの「つくる・まわす・とどける」に対応し、**実力の測定**から**継続的なプロジェクト運用**までを一気通貫で回す。

### フェーズ1：実力調査
メンバーがコーディングテストを解く → 終了と同時にAIメンターが「なぜこうした?」と壁打ち・深掘り → 書くプロセスの信号（打鍵/ペースト/実行）と回答の深さから **3軸スコア**を算出。メンバーには理解度・効率を、マネージャーには加えて信頼性（自力度）と講評を出し分ける。

### フェーズ2：プロジェクトマネジメント
マネージャーがプロジェクトをAIに投入 → フェーズ1のスコアを基に**作業量を割り当て**、壁打ちで割り振り → メンバーが遂行中もAIが**常時監視して制作過程を評価し、バグ・行き詰まりを早期発見**（DevOps「まわす」の核）→ 手直し申請・余剰作業の再分配 → 完成。GitHub連携で提出物を取り込む。

## スコア設計（3軸 → Dreyfus 5段階）

| 軸 | 意味 | 主な根拠 |
|---|---|---|
| **理解度**（習熟度） | 実力の段階そのもの | 「なぜこうした?」への回答の深さ（因果／代替／トレードオフ／限界認識） |
| **手を動かす効率** | 素早さ・無駄のなさ | コードの簡潔さ・直截さ、手戻り（実行回数）の少なさ |
| **信頼性**（自力度） | スコアを鵜呑みにしてよいか | ペースト率・打鍵リズム・説明の一貫性（丸写し兆候を検出） |

理解度を Dreyfus 5段階（初心者→上級初心者→一人前→中堅→達人）に変換し、信頼性が低い場合は「（要確認）＝スコアは出たが鵜呑み注意」の但し書きを添える。マネージャーは「高いが怪しい」を見抜ける。

**本番での実測例**（同じ提出コード・異なるプロセスと回答）:

| ケース | 理解度 | 効率 | 信頼性 | 判定 |
|---|---|---|---|---|
| 深い回答・自力（ペースト0） | 95 | 90 | 100 | 達人 |
| 中間（説明あっさり） | 45 | 60 | 60 | 上級初心者 |
| 丸投げ・ペースト多用 | 30 | 70 | 20 | 上級初心者（要確認） |

同じコードでも、プロセスと思考の深さで実力と自力度がくっきり分かれる。

## 主な機能（本番稼働）

**フェーズ1：実力調査**
- Monaco エディタでのコーディングテストと、書くプロセス（打鍵/ペースト/実行）の記録
- 提出直後のAIメンター壁打ち（「なぜこうした?」を一問ずつ深掘り、会話継続）
- Gemini の構造化出力による3軸スコアの実算出（上表のとおり本番で明確に差が出る）
- マネージャー評価一覧への反映 ＝ Dreyfus段階・信頼性の但し書き・ひとことで出し分け

**フェーズ2：プロジェクトマネジメント**
- プロジェクト要件を投入すると、AIが実力調査スコアを基に**作業を自動割り当て**（担当・作業量%・理由）
- 自力度が低いメンバーはレビュー必須タスクへ、理解度・自力度が高いメンバーは難所・コア設計へ、という原則で配分
- メンバー側に担当が届き、遂行中はAI監視のもとで進めて提出

## 技術構成

- **実行基盤**：Cloud Run（サーバーレスコンテナ、ゼロスケール）
- **AI**：Gemini API（`responseSchema` による構造化出力でスコア算出・作業割り当て／混雑時リトライ＆モデル退避）
- **フロント**：Next.js 16 ＋ Monaco Editor（青白・青枠のUI、ダーク/ライト対応）
