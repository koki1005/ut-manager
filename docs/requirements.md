# 要件定義：プロセス監視型・育成AIメンター「UT Manager」

DevOps × AI Agent Hackathon 2026 提出作品の要件定義。2026-07-08起票、07-09更新（主役をプロジェクト型に変更、スコア設計・デザイン方針・画面ラフ・機能の3階層仕分けを追加）。締切 2026-07-12(日)23:59、一人参加。名前は「UT Manager」に確定。

## 1行ピッチ
成果物ではなく「コードを書くプロセス」を見て、エンジニアの本当の実力を見抜き、マネージャーの人員配置を支えるAI。使うほど各人にパーソナライズされて育つ。

## 背景・課題
- AIの普及で「手を動かせる人」は激増したが、理解の伴う本物のエンジニアはむしろ減った。
- GitHub等の成果物は嘘をつける（コピペも自力も、コミットされれば同じ顔）。理解度は完成コードだけでは測れない。
- 見るべきは「どう手を動かしたか＝プロセス」。プロセスは嘘をつけない。
- ITはスキルで年収・就職がはっきり差別化される。実力を正しく可視化する価値は大きい。

## ターゲット
- 主役：**プロジェクト型**。開発チームのマネージャーと、そのメンバー。
- 入口：メンバーに **初回コーディングテスト** を解かせて実力のベースラインを取得。お題は **マネージャーが設定** できる。
- 拡張：教育型（学ぶ本人向け）、採用の習熟度診断。

## コアコンセプト（3本柱）
1. **プロセスを見る** — 打鍵・ペースト・手戻り・停止・実行回数・エラー履歴を記録。成果物でなく過程で判断。
2. **事実で測る** — 自己申告を信じず、行動シグナル＋「なぜこうした?」への回答の深さで実力を測る。
3. **育つ** — 各ユーザーの履歴（盛る/隠す癖、詰まりやすい領域）を蓄積し、使うほど判定と指導が個別最適化される。

## 習熟度スコア設計（2軸 → Dreyfus 5段階で出力）
既存基準を参照：CodeSignal（Developing→Expert可視化）、Codility（Code Playback＝過程の再生）、Dreyfusモデル（初心者→上級初心者→一人前→中堅→達人、スキル領域ごと）、SFIA（IT責任レベル7段階）、Grammarly Authorship（打鍵・ペーストを事実ログで判定）、AI code detector（スタイル・コメントで90%+、短いコードは50%）。

**重要な根拠**：2026年1月の研究で、行動ログだけでは「人がキーボードを操作したこと」までしか分からず「中身を人が考えたか」は分からないと証明（Copytype攻撃で99.8%回避）。→ 理解を測るには「思考を問う」しかない。この必然性が作品の新規性の芯。

スコアは2軸に分ける。
- **オーサーシップ軸（自力度）**：ペースト率、打鍵リズムの一定さ（丸写しは機械的に一定）、手戻りパターン、AIっぽさ（空白が綺麗すぎ・コメントがAI的）。
- **習熟度軸（理解度）**：「なぜこうした?」への回答の深さ、トレードオフ・捨てた選択肢を語れるか、罠質問への耐性、停止の質（考える停止か詰まる停止か）。

2軸を掛けて4象限（自力×理解=本物／自力×浅い=伸びしろ／貼る×説明できる=AI使いこなし／貼る×説明できない=丸投げ）。最終出力は Dreyfus 5段階に寄せて「この領域で"一人前"」と提示。

**量トラップの補正**：「たくさん書かせてボロを出す」は有効だが、量そのものを高評価にしない（本物は簡潔を選ぶので誤判定する）。評価するのは「量を増やさせた時に一貫性・密度を保てるか」。

### スコアの合成（重み付け・叩き台）
2軸の役割を分ける：**習熟度軸＝実力の段階そのもの／オーサーシップ軸（自力度）＝その判定をどれだけ信用していいか（信頼度）**。自力で書いてるほど実力スコアが信用でき、貼りまくりなら「スコアは出たが鵜呑み禁止」のフラグになる。

- **オーサーシップ軸（自力度, 0〜100）** ＝ 100 −（45×ペースト率 ＋ 25×打鍵の機械的一定さ ＋ 30×AIっぽさ）。ペースト率（貼った文字数÷全体）を最重視、丸写し兆候の打鍵一定さ、AIっぽさ（コメントの整いすぎ・空白）を加える。
- **習熟度軸（実力, 0〜100）** ＝ 50×回答の深さ ＋ 20×量トラップ耐性 ＋ 15×(1−詰まり停止度) ＋ 15×(1−検索依存度)。回答の深さに半分振る（行動ログでは理解を測れない＝思考を問うのが主役という研究根拠）。詰まり停止・検索依存は減点に効く。

**「回答の深さ」の定義（りこ設計）**：Geminiが「なぜこの実装?」の回答を4観点で各0〜1採点し平均する。理由を言えるか（因果）／捨てた選択肢に触れるか（代替）／計算量・可読性の天秤を語れるか（トレードオフ）／崩れるケースを分かってるか（限界認識）。この4観点はDreyfus段階とほぼ対応（達人は4つ全部、初心者は因果すら曖昧）。

**検索依存の補正**：検索単体では減点しない（調べるのは有能さでもある）。「検索直後に貼って説明できない（回答の深さが低い）」時だけ効かせる。検索は回答の深さと掛けて見る。

**Dreyfus 5段階への変換**：実力軸の点で段階を出す。85+=達人／70〜84=中堅／50〜69=一人前／30〜49=上級初心者／30未満=初心者。そこに自力度を信頼度として添える（例「一人前だが自力度42%、要確認」）。マネージャーはこの但し書きで「高いが怪しい」を見抜ける。

※重みの数値は実データ前の勘の叩き台。実際に解かせて調整する前提。審査では「重みが完璧」より「根拠を持って合成している」ことが効く。過学習で数字をいじり倒さない。

## MVP（4日で確実に動かす範囲）
プロジェクト型の入口（初回コーディングテスト）を1本、確実にデモできる状態まで。
1. **マネージャー画面**：お題を設定、メンバーの結果（2軸スコア・Dreyfus段階・割り当て提案）を見る。
2. **メンバー画面**：Web上のコードエディタ（Monaco）で設定されたお題を解く。
3. 書くプロセスをイベントで記録：打鍵速度・ペースト検知・手戻り・停止時間・実行回数・エラー。
4. 軽い判定はブラウザ側で即時、重い判断（理解度評価・質問生成）だけGeminiに投げる二段構え。
5. 節目でGeminiが「どうしてこの実装にした?」と問い、思考ロジックで理解度を見極める（深掘り可）。
6. プロセス＋回答から2軸スコア→Dreyfus段階と作業信頼性を根拠つきで提示。
7. マネージャーへ「この人にはこのレベルのタスクが適切」という **割り当て提案**。
8. 履歴を保存し、次回以降そのユーザー向けに判定が補正される（育つAIの芽）。

**デモの見せ場**：マネージャーがお題設定→メンバーが解く→AIが手の動きと回答から実力を暴いて割り当てを提案する一連。特に一発質問で本物度を暴く瞬間。

## デモ用サンプルお題（りこ選定・マネージャーが差し替え可能）
- **ウォームアップ**：文字列を受け取り、最も頻出する単語を返す関数（同数時の扱いで差が出る）。
- **本命（DevOpsテーマ整合）**：時系列ログ配列から「直近N分間のエラー率」を返す関数。掘りどころ＝ソートするか一度の走査か（計算量）、時刻境界の扱い、ログ0件時のゼロ除算をどうするか。エッジケースと設計判断で習熟度がくっきり出る。

## デザイン方針
- 参考：ConoHa VPS コンパネ、GitHub（どちらも白基調＋青アクセント＋情報を整理した静的UI）。
- 白基調に青いラインのアクセント。静的な画面。テーブル／カード中心で情報密度を持たせる。
- ダークモード／ライトモード両対応（GitHubが見本）。
- PC専用（レスポンシブ対応は不要）。

## 画面ラフ
メンバー画面（テストを解く側）：
```
┌──────────────────────────────────────────┐
│ お題：直近N分のエラー率を返す関数          │
├────────────────────────┬─────────────────┤
│   コードエディタ         │  AIメンター       │
│   (Monaco)             │ 「なぜソートを?」  │
│                        │  [回答を入力]     │
├────────────────────────┴─────────────────┤
│ [実行]  出力・エラー表示                    │
└──────────────────────────────────────────┘
```
マネージャー画面（結果を見る側）：
```
┌──────────────────────────────────────────┐
│ お題設定：[エラー率の関数 ▼] [編集]         │
├──────────────────────────────────────────┤
│ メンバー  実力(Dreyfus)  自力度  ひとこと    │
│ 佐藤      中堅          88%    信頼できる   │
│ 田中      一人前(要確認)  42%    貼り多い    │
│ 鈴木      上級初心者     75%    検索依存    │
├──────────────────────────────────────────┤
│ 割り当て提案：佐藤に難タスク、田中は要面談    │
└──────────────────────────────────────────┘
```
「ひとこと」欄＝傾向分析（サボり/丸投げ/真面目）を1行で。

## 目玉デモ拡張（コア完成の直後に足す・最優先の追加）
- **画面の間引き監視**：Screen Capture APIで数秒〜十数秒おきに1枚キャプチャ→**画像から文字だけ抽出（OCR／エディタなら直接テキスト取得）してGeminiに推論させる**（画像で投げず軽量化）→判定テキストだけ残し元データは即破棄（エフェメラル・データ最小化）。
- こうきの「必要な部分だけ抽出して即破棄」を実装。監視の倫理を突かれた時の最強の回答になる。

## 将来性の階段（発表で語る拡張ビジョン）
1. いま(MVP)：初回テストのプロセスを見て実力を測り、割り当てを提案。
2. 次：実プロジェクトの日々のコーディングを継続監視し、作業信頼性と進捗の真偽を検証。
3. その先：デスクトップ全体の作業を必要な瞬間だけ抽出・即破棄で見守り、社内連携を最適化。
4. 究極：要件定義を壁打ちで固め→仕様書を自動生成→チームへ仕事を割り当てまで一気通貫。

## 技術構成（ハッカソン必須要件マッピング）
- **実行基盤（必須①）**：Cloud Run（コンテナを載せるサーバーレス、ゼロスケール・実行時間課金）。
- **AI技術（必須②）**：Gemini API。まず「ダイレクト×generateContent」で即スタート、思考ステップの可視化に Interactions API を検討。
- **検索・記憶**：Elastic。プロセスログの蓄積・検索・照合、各ユーザーの長期記憶。スポンサー加点＋DevOpsテーマ適合。
- **フロント**：Next.js ＋ Monaco Editor。デプロイは Vercel か Cloud Run。
- 画面監視は画像でなくテキスト抽出後に推論（軽量・低コスト）。
- Google Cloud $300クレジット活用。全部クラウド処理でローカルスペック不問。

## スコープ（機能の3階層仕分け）
### MVP（確実に作る）
- マネージャー：お題設定、メンバー一覧＋スコア表示、割り当て提案。
- メンバー：エディタで初回テスト、プロセス監視、なぜ?質問、2軸スコア→Dreyfus表示。
- デザイン：白+青、ダーク/ライト、PC専用。
- 履歴パーソナライズの芽。

### 格上げ候補（コアが動いたら足す・軽い）
- 傾向分析：マネージャー画面の"ひとこと"1行（サボり/丸投げ/真面目）。スコアの副産物で軽い。
- ランキング／エンジニアとしての位置付け：スコアを並べるだけ、モチベ要素でデモ映え。
- 画面の間引き監視：目玉デモ。

### 将来（発表で語る・今は作らない）
- 継続教育モード（弱点に課題を出し続ける学習ループ）。
- 実タスクのレビュー・お直し・プルリクエスト。
- 教育実施率、要件定義モード、プロジェクトマネジメントモード（各々別アプリ級に重い）。
- 採用診断、自動GitHubプッシュ、外部AI利用の有意義さ判定、デスクトップ全体監視。

**線引きの理由**：MVPは「実力を見抜いて配置する」一本の筋を完璧に通すことに全振り。機能を広げるより一本が動いて説得力を持つ方が、審査でも実装の現実でも強い。

## 審査基準への対応
1. AIエージェントが中核・自律性 → プロセス監視→判断→質問→スコア→割り当て提案の自律ループ。
2. 課題設定・新規性 → 「AI時代に本物のエンジニアをどう見抜き適材適所に置くか」。行動ログでは理解を測れないと証明された今、思考を問うプロセス監視に必然性。
3. ユーザビリティ → マネージャー／メンバーの2画面、スコアの根拠表示。将来 A2UI 検討。
4. 実用性・体験価値 → 実力を事実で暴き、人員配置に直結させる体験。
5. 実装力 → 必須ツールの一直線構成、間引き＋テキスト抽出＋即破棄のプライバシー設計、拡張の階段。
